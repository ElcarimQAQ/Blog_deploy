架构

离线处理：数据提取->数值计算->结构分析->存数据库
在线查询：问题分析（问题分级、要素提取）->答案召回（正则匹配、ES检索、NL2SQL）->LLM拼装->答案格式化（后处理）
Router: 问题分类， 关系到下一阶段输入

分类标准

  • type1：可以单次完成的客观基础查询
    • type11，问一个单项存在于数据库内容的题目， 可以直接完成的客观基础查询。如：2020年一品红药业股份有限公司其他非流动金融资产是多少元？
    • type11-2，可以直接完成的隐含类基础查询，如法定代表人是否相同的题目，赛方定义为type2，本方案定义为11。如：南京寒锐钴业股份有限公司2021年的法定代表人与2019年的法定代表人相比相同吗？
    • type12，统计题目。如：2019-2021年哪些家上市公司货币总额均位列前十？
  • type2：使用type1结果进行组合后的客观高阶查询
    • 人话： 计算题， 根据sql查询出来的结果计算
    • type21，特定型公式，如速动比率这样的公式。在直接完成的客观基础查询之上再进行数学运算的查询，数学运算的公式属于特定公式，不具有广泛推导的性质。如：2019年贵研铂业股份有限公司速动比率为多少？保留两位小数
    • type22，在直接完成的客观基础查询之上再进行数学运算的查询，数学运算的公式固定。如增长率，其实就是type11的一个组合。如：在2020年的财务数据中，新华文轩管理费用增⻓率是多少？保留两位小数
  • type3代表需要模型使用(type31)/不使用相关资料(type32)进行主观总结的查询
    • type31：需要结合财报知识进行总结回答的无精确答案问题。如：根据2019年的年报数据，哈空调研发投入的情况，请做简要分析。
    • type32：知识理解的无精确答案问题。如：什么是其他债权投资？

n2sql：问题转化为查询， 较好的理解问题意图
normalize：规范化回答

DocTree

  • 年报的格式具有较高的一致性，需要的答案内容，如三表数据，基本信息等，都在特定章
    节的特定大标题之下，使用树的结构进行存储可以有效剪枝搜索范围，快速搜索到想要的答案。
  • type31类问题需要找到文档中的特定段落，通过建树可以对树的中间节点（即各级标题）进行
    高效针对的搜索，找到想要的段落。此外搜索标题相比于搜索正文更加适配于向量化搜索 ，避免无关内容挤占召回位置。

build_tree


find_pattern: 正则匹配，确定Node_type
文本是否为数字，如果是，返回-2和空字符串。
然后，它会使用正则表达式匹配tables_pattern所定义的模式，如果成功匹配，则返回6和匹配到的字符串。
接着，它会使用一系列预定义的正则表达式模式（dirty_patterns）尝试匹配文本，并返回第一个匹配到的字符串和-2。
最后，它会使用一系列正则表达式模式（patterns），尝试在文本中寻找匹配项。如果找到，则返回该匹配项的索引和匹配到的字符串。
如果找不到匹配项，则返回-1和空字符串。

Normalize:



translate_sql，它接受一个字符串 sql 作为输入参数，然后对其中的中文进行转化为拼音。
函数首先使用正则表达式 [\u4e00-\u9fa5]+ 匹配出字符串 sql 中的所有中文字符。然后，它会排除掉中文字符前后可能存在的单引号和百分号，将中文字符添加到 reps 列表中。接着，它将 reps 列表去重，按照字符串长度降序排序，然后遍历列表中的每个中文字符串，将其转化为拼音，使用拼音替换原 sql 字符串中的相应中文字符。最后，函数返回替换后的 sql 字符串。
